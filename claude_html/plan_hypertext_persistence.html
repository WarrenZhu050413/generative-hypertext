📋 Task Plan: Hypertext Persistence & Save Flow

Objective

Enable generated hypertext to persist between page reloads and give users a way to export/import the current annotated HTML state.

Approach

Introduce a client-side persistence layer with range serialization, hook session mutations into durable storage, and add explicit save/export UX for portability.

Complexity: Medium-High
Verification Hash: eaf38f05f7958876
⚡ Implementation Steps ▶
  1. Step 1: Audit session lifecycle and state mutation points
    File: hypertext/hypertext-experience.js
    Review how sessions are created (`sessions` map around createHypertextExperience) and where `session` fields mutate (`runHypertext`, `updateWrapperWithResult`, tooltip geometry handlers) to catalogue every spot that must persist data.
  2. Step 2: Design persistence schema & range serialization
    Files: hypertext/hypertext-experience.js, new helpers in hypertext/
    Define a JSON schema capturing subject text, sanitized selection HTML, assistant result payloads, tooltip geometry, pin state, timestamps, and a DOM anchor (e.g., XPath/CSS selector + text offsets). Decide how to version entries for future migrations.
  3. Step 3: Implement storage service
    New: hypertext/session-persistence.js (or similar)
    Build a thin module exposing `loadSessions()`, `saveSessions(sessions)`, `upsertSession(session)`, `removeSession(id)`. Prefer `localStorage` initially with batched writes (debounce/throttle), but architect an interchangeable backend (IndexedDB) if state grows.
  4. Step 4: Instrument write-through persistence hooks
    File: hypertext/hypertext-experience.js
    After session creation, updates in `runHypertext`, `updateWrapperWithResult`, pin toggles, tooltip drag/resize, and chat message edits, call the persistence module. Ensure DOM-only fields (like `wrapper`) are excluded before serializing.
  5. Step 5: Rehydrate sessions on load
    Files: hypertext/hypertext-experience.js, new persistence module
    During `createHypertextExperience` initialization, load saved entries, validate anchors against the current DOM, rebuild wrapper nodes (insert spans with `dataset` attributes), repopulate the `sessions` map, and restore tooltip state without triggering fresh generations.
  6. Step 6: Add explicit Save/Load UX
    File: hypertext/hypertext-experience.js (palette & tooltip UI)
    Expose UI controls (e.g., “Save hypertext to disk”, “Import saved hypertext”) that serialize the current state to downloadable JSON/HTML via Blob + `URL.createObjectURL`, and allow restoring from user-selected files for cross-device use.
  7. Step 7: Implement cleanup & versioning safeguards
    Files: hypertext/session-persistence.js, README.md
    Add mechanisms to clear invalid anchors, prune stale entries, and bump schema versions when selector matching fails. Document the storage footprint and provide a “Reset saved hypertext” action.
✓ Testing & Validation ▶
  • Manual Reload: Generate multiple hypertext pills, reload the page, confirm pills, tooltips, and chat history rehydrate in place.
  • DOM Drift Check: Modify surrounding page content slightly, reload, ensure anchors either gracefully adjust or the entry is flagged for user attention.
  • Export/Import: Save a snapshot, clear storage, import the file, and verify the state reinstates identically.
  • Quota & Performance: Stress-test with high volume of sessions to observe storage size, write frequency, and restore latency.
📍 Prerequisites & Context ▶
  • Hypertext runtime: Understand `createHypertextExperience` initialization and how selection caching currently works (`cacheSelection`, palette interactions).
  • Session data shape: Document existing `session` fields (messages, lastResult, geometry) to avoid serializing DOM references.
  • Target environments: Confirm the host pages (extension vs demo) allow `localStorage`/`IndexedDB` and handle CSP constraints for Blob downloads.
  • Existing UX patterns: Review tooltip/palette UI so new controls match styling and keyboard shortcuts.
⚠️ Potential Issues & Mitigations ▶
  • Anchor Drift: Page edits between saves may break selectors; mitigate with text-context matching and graceful fallback messaging.
  • Storage Limits: `localStorage` caps (~5MB); compress or migrate to IndexedDB if serialized chat threads become large.
  • Security & Privacy: Generated content may include sensitive data—consider opt-in persistence, encryption, or workspace scoping.
  • Concurrency: Multiple tabs editing the same document may stomp saved state; leverage `storage` events or instance IDs to reconcile.
  • Performance: Frequent write-through can jank; batch updates and avoid serializing large DOM fragments synchronously.
📊 Post-Implementation ▶
  • Documentation: Update README/demo instructions explaining persistence toggles, export/import workflow, and recovery options.
  • Analytics: Instrument optional usage metrics to learn how often persistence is used and detect failure rates (if acceptable for product).
  • Future Enhancements: Explore cloud sync (user accounts) or shareable permalinks once local persistence proves reliable.
  • Maintenance: Establish schema migration strategy and automated cleanup jobs for outdated entries.
