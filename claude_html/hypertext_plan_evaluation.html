Overview

From my own experience, I reviewed docs/research/planning/llm_generated_hyperlinks_on_the_fly.html which imagines highlight-driven child card generation with streaming feedback.

Strengths

Looking at this differently, the plan aligns with the product vision of instant knowledge scaffolding and reuses existing card creation plus connection storage.

  • Flow: Cmd+L shortcut and context menu keep the action under three seconds.
  • Reusability: The proposed textSelection.ts helper can also serve quote extraction and expandable links.
  • Delight: Streaming feedback and emanation animation reinforce the stretchtext story.

Constraints

The problem is direct DOM wrapping via range.surroundContents inside React-managed nodes can desynchronize markup and break future renders.

  • State sync: We may need to render highlights via state rather than raw DOM mutation.
  • Undo: The plan should define how to revert temporary spans if generation is cancelled.
  • Storage: Generated children need metadata about their source selection for traceability.

Feasibility Notes

I am curious whether we can centralize LLM calls through the existing ChatService so we do not invent yet another request layer.

Share the selection manager between cards inside the side panel and canvas so we test once and reuse.
ðŸŒ± Expandable Parent-Child Cardsâ–¶
High impact Medium-high complexity Estimate: 8-12h

Overview

From my own experience, I explored docs/research/planning/expandable_parent_child_cards.html which brings Ted Nelson style stretchtext onto the canvas.

Strengths

Looking at this differently, the proposal extends existing card relationships without changing the mental model for parent and child nodes.

  • Model: Additional fields like childCardIds and layout metadata plug directly into current card records.
  • Layouts: The radial/grid/tree options give us flexibility to match content density.
  • Experience: Inline spans with icons can make every paragraph feel alive.

Constraints

The problem is storing startOffset and endOffset will drift once the parent content is edited or reformatted.

  • Editor path: Card content currently flows through Markdown; raw HTML spans need sanitization review.
  • Sync: Layout calculations must coexist with manual card positioning so we should allow users to toggle back to freeform layout.
  • Accessibility: Ensure expand links are focusable and keyboard operable.

Feasibility Notes

I wonder if we can store anchor markers as lightweight shortcodes (e.g. [[expand:child-1|alignment]]) and render spans at runtime to keep Markdown authorship smooth.

Prototype the expansion toggle using existing React Flow fitView helpers before committing to the full animation stack.
ðŸ§­ Hypertext Navigation Featuresâ–¶
High impact Low-medium complexity Estimate: 4-8h for first milestones

Overview

From my own experience, I reviewed docs/research/planning/hypertext_navigation_features.html alongside docs/research/planning/plan_bidirectional_links.html which focus on quote extraction, backlinks, and citation jumps.

Strengths

Looking at this differently, these upgrades lean on data we already have: connection records, message parents, and React Flow navigation.

  • Quote cards: Selection to quote card workflow reuses connection creation flows.
  • Backlinks: The proposed BacklinksPanel slots neatly into existing card chrome.
  • Citation jump: fitView and highlight routines already exist for card focusing.

Constraints

The problem is selection offsets for quotes share the same drift risk as expandable links, so we should anchor highlights using string matching with tolerance.

  • Performance: Loading backlinks on every render needs memoization or caching.
  • Noise control: Generated-from links may overwhelm the panel; we might filter by connection type.
  • Outline view: Detecting hierarchies will require metadata we do not yet track.

Feasibility Notes

I am curious about batching the upcoming Playwright tests so quote extraction, backlinks, and hyperlink generation share one regression suite.

Deliver backlinks and quote extraction together to prove the navigation loop before adding inline citation jumping.
