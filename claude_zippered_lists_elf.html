Why "Zippered"? Like a zipper, the links connect corresponding "teeth" (entries) from two separate tracks (lists), and moving the fabric (reordering) doesn't break the zipper connections.

Three Elements

  • Entry: A discrete unit of information (text, image, command)
  • List: An ordered set of entries; one entry can be in multiple lists
  • Link: A 1-for-1 connector between entries in different lists

Key Properties

  • No hierarchies: User defines all relationships
  • Permutation-invariant: Reordering doesn't break links
  • Evolutionary: Easy to create variants/versions

Patching Technique â€” How variants are stored efficiently â–¶
Problem: If you want to create many versions/variants of a manuscript, storing full copies would be wasteful.

Solution: Patches (like diff/patch in Unix)

Instead of storing full copies, the ELF stores:

  • Original: Full content of version 1
  • Variant: Pointer to original + list of changes (patches)
Version 1: "The quick brown fox jumps"
Version 2: [ptr to v1] + [PATCH: replace "brown" â†’ "red"]
Version 3: [ptr to v2] + [PATCH: insert "very" before "quick"]

This is essentially what Git does with commits (delta compression).

Philosophical Implications

"The only ultimate structure is change itself" â€” Nelson's radical insight â–¶

Why Fixed Categories Fail

From the paper (p. 97):

"Human ideas, science, scholarship and language are constantly collapsing and unfolding. Any field, and the corpus of all fields, is a bundle of relationships subject to all kinds of twists, inversions, involutions and rearrangement... While the disappearance and up-ending of categories and subjects may be erratic, it never stops."
Modern validation:
  • Dewey Decimal System: Created 1876, now inadequate for digital/internet topics
  • Library of Congress: Struggles with interdisciplinary fields
  • Wikipedia categories: Constantly reorganized by editors
  • Folder hierarchies: Break down as projects evolve (hence tags/search)

Nelson's Solution: Evolutionary Structures

Information systems must:

  • Accept any arrangement users impose (no built-in hierarchies)
  • Support rearrangement without data loss
  • Allow multiple simultaneous organizations (tags > folders)
  • Preserve history of how categories evolved

This is why the ELF has NO predetermined structure â€” it's a "multifarious, polymorphic, many-dimensional, infinite blackboard."

Why This Matters Today

Nelson Was Right About:

  • Hypertext: The web proved we need interconnected documents
  • Versioning: Git is now essential for code/documents
  • Non-hierarchical organization: Tags beat folders for search
  • Visual arrangement: Spatial canvases (Miro, Figma, your project!) are powerful
  • Bidirectional links: Roam, Obsidian, Notion implement this

What We Still Haven't Fully Implemented:

  • Transclusion: Including same content in multiple contexts without copying
  • Parallel documents: Comparing equivalent sections across versions
  • Dynamic outlining: Outline changes automatically reorder content
  • Full bidirectionality: Web links still break; no universal backlinks
  • Unified system: We use Git for code, Google Docs for writing, Notion for notes â€” Nelson wanted one system
