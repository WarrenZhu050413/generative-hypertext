‚ñ∂Navigation browser_navigate - Go to any URL

Purpose

Navigate the browser to a specific URL. This is typically the first tool you'll use to visit a webpage.

When to Use

Examples:
  • Testing your Chrome extension on a real website
  • Visiting a page where you're already logged in

Input Schema

Parameter Type Required Description

Output

Returns:

  • Console messages (logs, errors, warnings)
  • Page state (URL, title, snapshot)
  • Page snapshot (YAML structure of visible elements)

Example Usage

  url: "https://example.com"

‚ñ∂Interaction browser_click - Click on elements

Purpose

Click on any element on the page (buttons, links, inputs, etc.)

When to Use

Examples:
  • Clicking buttons to test functionality
  • Opening dropdown menus
  • Following links
  • Activating interactive elements

Input Schema

Parameter Type Required Description
element string ‚úì Human-readable element description
ref string ‚úì Element reference from page snapshot (e.g., "e6")
button string "left" (default), "right", or "middle"
doubleClick boolean Perform double-click instead of single
modifiers string[] ["Alt", "Control", "Meta", "Shift"]

Example Usage

// First, get a snapshot to see available elements
mcp__playwright__browser_snapshot({})

// Then click on an element using its ref
mcp__playwright__browser_click({
  element: "More information link",
  ref: "e6"
})

‚ñ∂Inspection browser_snapshot - Get page structure

Purpose

Get a structured YAML representation of all elements on the page. This is essential before you can interact with elements.

When to Use

Always use this first!
  • Before clicking, typing, or hovering
  • To find element references (refs) for interaction
  • To understand page structure
  • To verify elements exist

Input Schema

No parameters required

Output

Returns a YAML snapshot showing:

  • All interactive elements
  • Element types (button, link, textbox, etc.)
  • Element text content
  • Element refs (e.g., "e6", "e12") - needed for interaction
  • Element attributes (cursor, URLs, etc.)

Example Usage

mcp__playwright__browser_snapshot({})

Example Output

- generic [ref=e2]:
  - heading "Example Domain" [level=1] [ref=e3]
  - paragraph [ref=e4]: This domain is for use...
  - paragraph [ref=e5]:
    - link "More information..." [ref=e6] [cursor=pointer]:
      - /url: https://www.iana.org/domains/example

Workflow:

1. Snapshot               2. Find Ref          3. Interact

‚ñ∂Inspection browser_evaluate - Run JavaScript code

Purpose

Execute custom JavaScript code in the page context. Very powerful for custom inspections.

When to Use

Examples:
  • Check if extension globals exist (window.__nabokobClipper__)
  • Query Shadow DOM elements
  • Get computed styles
  • Access browser APIs not exposed in snapshot

Input Schema

Parameter Type Required Description
function string ‚úì JavaScript function as string: "() => { ... }"
element string Element description (if evaluating on element)
ref string Element reference (if evaluating on element)

Example Usage

// Check if extension loaded
mcp__playwright__browser_evaluate({
  function: "() => { return typeof window.__nabokobClipper__ !== 'undefined'; }"
})

// Count Shadow DOM elements
mcp__playwright__browser_evaluate({
  function: "() => { return Array.from(document.querySelectorAll('*')).filter(el => el.shadowRoot).length; }"
})

// Get element property
mcp__playwright__browser_evaluate({
  function: "(element) => { return element.getBoundingClientRect(); }",
  element: "Main container",
  ref: "e5"
})

üí° Tips & Best Practices

‚úÖ Do's

  • Always snapshot first - Get page structure before interacting
  • Check console regularly - Catch errors early
  • Use browser_evaluate - For custom checks not available in snapshot
  • Take screenshots - Visual verification is powerful
  • Wait for dynamic content - Use browser_wait_for when needed

‚ùå Don'ts

  • Don't click without snapshot - You need the ref first
  • Don't ignore console errors - They reveal issues
  • Don't assume timing - Wait for elements/text when needed
  • Don't hardcode refs - They change, always get fresh snapshot
  • Don't skip verification - Check results after actions

üéì Key Takeaways

Essential 3-Step Pattern

  1. Snapshot - Get page structure with refs
  2. Interact - Use refs to click/type/etc
  3. Verify - Check console/screenshot results

Most Used Tools

  • browser_navigate - Go to pages
  • browser_snapshot - Get structure
  • browser_click - Interact
  • browser_console_messages - Debug
  • browser_take_screenshot - Verify
